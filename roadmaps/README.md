# Roadmaps

Product roadmaps and feature planning for InMoney.

## Contents

- [Current Quarter](./current.md) - Active roadmap
- [Backlog](./backlog.md) - Future features backlog
- [Archive](./archive/) - Past roadmaps

## Roadmap Structure

Each roadmap document should follow this format:

```markdown
# Roadmap: [Period]

## Overview
Brief summary of goals for this period.

## Goals
- [ ] Goal 1
- [ ] Goal 2
- [ ] Goal 3

## Features

### Feature Name
- **Status**: Planning | In Progress | Completed | Deferred
- **Priority**: P0 (Critical) | P1 (High) | P2 (Medium) | P3 (Low)
- **Target**: [Date or Sprint]
- **Owner**: @username
- **Description**: What this feature does
- **Success Criteria**:
  - Criterion 1
  - Criterion 2
- **Tasks**:
  - [ ] Task 1
  - [ ] Task 2

## Retrospective
(Added at end of period)
- What went well
- What could improve
- Key learnings
```

## Status Definitions

| Status | Description |
|--------|-------------|
| Planning | Being scoped and designed |
| In Progress | Actively being developed |
| Completed | Shipped and verified |
| Deferred | Moved to future roadmap |
| Cancelled | No longer planned |

## Priority Levels

| Priority | Description | Timeline |
|----------|-------------|----------|
| P0 | Critical | Must ship this period |
| P1 | High | Should ship this period |
| P2 | Medium | Nice to have this period |
| P3 | Low | If time permits |

## Process

1. **Planning**: Define features at start of period
2. **Tracking**: Update status weekly
3. **Review**: Mid-period check-in
4. **Retrospective**: End-of-period review
